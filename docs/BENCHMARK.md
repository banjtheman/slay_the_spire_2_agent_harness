# Benchmark Design

This project can support two related benchmarks, but they should be scored
separately.

## Tracks

### Track A: Stateless Model Progression

Use this to compare models and local inference backends.

The harness is fixed. The system prompt, action schema, memory format, bridge,
tool permissions, retry policy, and route strategy are all pinned by the
benchmark version. The model receives only the current game state plus in-run
memory generated by the runner. It may not edit its own profile, scripts, or
encounter notes between runs.

Good for:

- Codex vs Claude Code vs Cursor Agent vs Ollama vs MLX
- local model comparisons
- prompt/model changes with controlled harness behavior
- reproducible progress metrics

### Track B: Adaptive Harness Progression

Use this to compare agent builders and harness strategies.

The submitted artifact is an agent profile or harness bundle. It may include
custom `AGENTS.md`, `ENCOUNTERS.md`, wrapper code, tool policies, local model
choice, and optional between-run learning under a declared budget.

Good for:

- "build the best Spire agent" competitions
- self-improving harness experiments
- research on long-running agents and online adaptation

Do not rank Track A and Track B together. Track A is a model evaluation. Track B
is a systems/harness evaluation.

## Recommended First Public Benchmark

Start with Track A.

It is easier to explain, easier to reproduce, and less gameable. Track B can
come later once the event format and trace schema are stable.

Suggested first config:

```yaml
benchmark_id: sts2_visible_v0
track: stateless_model_progression
game_version: recorded per run
bridge_patch_version: patches/cli-anything-sts2-bridge-compat.patch sha256
character: IRONCLAD
ascension: 0
seeds:
  public_dev: 5
  private_eval: 20
max_actions_per_run: 500
max_agent_seconds_per_action: 180
max_wall_clock_minutes_per_run: 90
tool_policy: no shell, no web, no file writes
memory_policy: in-run memory only
profile: agent_profiles/codex_base
scoring_version: v0
```

## Metrics

Store raw metrics and compute one headline score.

Core progress metrics:

- `victory`: `1` if the run wins, else `0`
- `act_reached`
- `floor_reached`
- `bosses_defeated`
- `elites_defeated`
- `rooms_completed`

Quality metrics:

- `final_hp`
- `max_hp`
- `gold_spent`
- `cards_added`
- `cards_removed`
- `potions_used`
- `relics_collected`

Agent cost and speed:

- `input_tokens`
- `cached_input_tokens`
- `output_tokens`
- `reasoning_output_tokens`
- `estimated_cost_usd`
- `agent_seconds`
- `actions_taken`
- `invalid_actions`
- `bridge_recoveries`
- `timeouts`

Suggested headline score for v0:

```text
score =
  10000 * victory
  + 1000 * bosses_defeated
  + 100 * floor_reached
  + 25 * elites_defeated
  + final_hp
  - 50 * invalid_actions
  - 100 * timeouts
```

This keeps "wins" above deep losses, while still differentiating failed runs by
progress and reliability.

```json
  "character": "IRONCLAD",
  "ascension": 0,
  "seed": "private_001",
  "score": 1245,
  "victory": false,

  "floor_reached": 27,
  "bosses_defeated": 1,
  "elites_defeated": 3,
  "actions_taken": 312,
  "invalid_actions": 4,
```

Keep JSONL traces as artifacts, but leaderboard rows should use compact summary
JSON so the table stays fast.

## Hugging Face-Style Leaderboard

A Hugging Face leaderboard shape works well for the public display:

```text
Public Gradio Space
  -> submissions dataset
  -> evaluator runner
  -> results dataset
  -> public leaderboard table
```

For this game, the evaluator should probably not run inside a public hosted
Space. Slay the Spire 2 requires a licensed local game install and a GUI-capable
runtime. Instead:

1. Use a public Space for the leaderboard UI and submission form.
2. Store submissions and results in Hugging Face datasets.
3. Run the actual evaluator on one or more trusted local machines with STS2 and
   the bridge installed.
4. Have the evaluator push signed result summaries and trace artifacts back to
   the results dataset.

This keeps the public leaderboard simple while respecting the fact that the game
itself is not a normal hosted benchmark dependency.

```json
  "notes": "uses curated encounter notes and local planner"
}
```

## Anti-Gaming Rules

Minimum rules for a meaningful leaderboard:

- Pin benchmark version, bridge patch hash, action schema, and scoring version.
- Use private eval seeds for official ranking.
- Publish public dev seeds for debugging.
- Separate model-only and adaptive harness tracks.
- Record full traces for audits.
- Disallow external web/search in Track A.
- Declare tool permissions for every result.
- Declare whether inference is local, hosted API, or mixed.
- Reset game/run state between seeds.
- Do not allow benchmark-specific seed memorization in submitted prompts.

## Implementation Plan

Phase 1:

- Add a `benchmark_runner` CLI that runs a fixed seed list and writes one summary
  JSON per run.
- Add deterministic result extraction from run memory and final bridge state.
- Add a `results/` directory schema and example fake results.
- Add Ollama/MLX harnesses as normal harness adapters.

Phase 2:

- Build a Gradio leaderboard Space backed by a Hugging Face results dataset.
- Add a submission dataset for model/backend/profile requests.
- Add a local evaluator worker that pulls pending submissions and pushes results.

Phase 3:

- Add Track B with bundle submissions, declared learning budgets, and separate
  rankings.
- Add run replay views using trace JSONL artifacts.
- Add per-encounter breakdowns and failure-mode dashboards.
